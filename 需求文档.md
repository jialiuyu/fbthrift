# ub.mem 用户态双边通信库 V1 需求说明

## 1. 背景

前期 ub.mem / shared-memory transport 验证已经证明该方向具备继续迭代的价值。现有工作主要围绕 `fbthrift` Rocket benchmark、`folly` transport 基础设施、`shm_poller` / `shm_directpoll` 路径和 socket baseline 对比展开。下一阶段项目目标不再只是把某条 RPC benchmark 跑通，而是沉淀一个独立的 C++ 用户态双边通信库。

V1 需求的核心问题是：在机架内跨机器、共享 ub.mem 内存域可用的前提下，交付一个可握手、可多 lane 扩展、可 benchmark、可被真实 RPC adapter 消费的通信库雏形。该库的数据面不走内核 socket，控制面使用 TCP bootstrap 完成跨机器建链和参数交换。

## 2. V1 一句话目标

V1 交付一个独立 C++ 通信库，通过 TCP 建立 rack-local peer 控制连接，协商固定 multi-lane ub.mem / shared-memory 数据面资源，并提供有界 slot-based、unordered atomic message 通信能力。库显式暴露 buffer ownership 和 completion 语义，同时提供 thin `Channel` wrapper，支撑独立 benchmark 和两个最小 e2e RPC adapter proof。

## 3. 设计原则

1. **独立库优先**：V1 以独立通信库为交付主体；adapter 只作为 e2e proof 和早期消费者，不能反向定义或绑死核心 API。
2. **数据面用户态**：应用 payload 只走 ub.mem / shared-memory data plane，不走 TCP 数据路径。
3. **控制面可用**：V1 必须提供 TCP bootstrap handshake。纯静态配置或脚本式 attach 不算可接受 V1 结果。
4. **消息而非字节流**：V1 不复制 TCP 的无界 byte stream 语义，而是定义有界 atomic message transport。
5. **资源边界显式**：slot、buffer、queue 都是有界资源，`NoBuffer` / `QueueFull` 是正常 backpressure。
6. **性能语义可见**：Core API 暴露 buffer ownership、completion、lane 和 poll 语义，避免把性能责任隐藏在 socket-like API 后面。
7. **演进空间明确**：V1 压住 dynamic lane、用户内存注册、复杂负载均衡和完整 URMA 兼容，但对象模型不能堵死这些后续方向。

## 4. V1 范围

### 4.1 In Scope

- 独立 C++ library。
- TCP bootstrap control plane，面向机架内跨机器 peer。
- ub.mem / shared-memory user-space data plane。
- fixed multi-lane，握手时协商 `lane_count`，运行时不动态增减 lane。
- lane-local `BufferPool`。
- bounded slot-based atomic message transport。
- API 层 unordered message 语义。
- library-managed buffer pool。
- explicit buffer ownership 和 completion。
- Core non-blocking poll API。
- thin `Channel` wrapper。
- header integrity validation 必选，payload checksum 可选。
- 独立 benchmark 和最小功能测试。
- `netpoll + Kitex/Keitex` 最小 e2e adapter proof。
- `fbthrift + folly` 最小 e2e adapter proof。

### 4.2 Out of Scope

- socket-compatible byte stream。
- API 层 FIFO 或跨 lane 全局有序消息。
- dynamic lane 扩缩容。
- arbitrary user memory registration。
- 完整 URMA / RDMA verbs 兼容。
- 生产级 RPC transport feature parity。
- RPC streaming、TLS/认证/加密、service discovery、跨机架路由和完整 fallback 语义。
- payload 自动 TCP fallback。
- retransmission、ACK repair、path failover 或 kernel-style failure recovery。
- 硬件级 ub failure、machine failure、silent platform failure 的软件恢复。

### 4.3 第一迭代需求分解

第一迭代的目标是交付一个可独立运行、可握手、可收发、可回收、可观测、可 benchmark、可被真实 RPC 栈调用的最小 ub.mem 双边通信库闭环。该迭代不追求完整 URMA 兼容，也不追求生产级 RPC transport feature parity；重点是把控制面、数据面、Completion 语义和资源生命周期定型，用独立 benchmark 证明 fixed multi-lane 的基础可行性，并用两个最小 e2e adapter 证明该库能进入真实 RPC 依赖链路。

优先级定义：

- `P0`：第一迭代必须完成；缺失则 V1-MVP 不成立。
- `P1`：第一迭代应完成；可以在不破坏 P0 闭环的前提下按风险裁剪，但必须记录原因。
- `P2`：第一迭代只需预留接口或写清约束；实现可进入后续迭代。

| 优先级 | 类型 | 需求项 | 综述 | 验收标准 |
|---|---|---|---|---|
| P0 | 架构/边界 | 独立库骨架 | 建立独立 C++ library，adapter 不得成为唯一使用路径。 | 可以构建独立库目标；示例程序不依赖 `fbthrift` 即可创建 `Context`、`Endpoint` 和 `Connection`。 |
| P0 | E2E/立项 | 双 adapter e2e proof | 第一迭代必须选择两条真实 RPC dependency path 接入，证明库不仅能跑 micro benchmark。 | `netpoll + Kitex` 和 `fbthrift + folly` 均能启动 client/server；最小 request/response 的应用 payload 通过 ub.mem 数据面传输；TCP 只用于 bootstrap/control。 |
| P0 | 控制面 | TCP bootstrap handshake | initiator / acceptor 通过 rack-local TCP 交换版本、peer、lane、slot、region 和 feature 信息。 | 正常握手进入 `Ready`；版本不兼容、lane/slot 参数不支持、region attach 失败时返回结构化 reject reason。 |
| P0 | 控制面 | TCP control connection 保持打开 | 握手后保留 TCP control connection，用于 close、error、heartbeat 和后续降级协商，不承载应用 payload。 | `graceful close`、peer reject、control connection close 可被本地 poll 到 `ControlCompletion` 或等价 control event；payload 不会 fallback 到 TCP。 |
| P0 | 数据面 | fixed multi-lane attach | 握手协商固定 `lane_count`，每条 lane 独立维护 slot space、ready queue、reclaim queue 和 lane-local buffer pool。 | `lane_count=1,2,4,8,16` 均可 attach；不支持运行时增减 lane；不支持 outstanding message 迁移。 |
| P0 | 数据面 | bounded slot-based atomic message | 每条消息占用 committed slot，receiver 要么看不到消息，要么拿到完整消息；API 层不承诺顺序。 | 并发发送下 receiver 不读取未 commit slot；跨 lane 到达顺序不作为验收；应用需要顺序时可使用自带 sequence number。 |
| P0 | 数据面 | ready / reclaim queue 语义 | ready queue 表达“对端有完整消息可读”；reclaim queue 表达“对端已释放 slot，发送方可复用”。 | `RecvCompletion` 由 ready queue 触发；`ReclaimCompletion` 由 reclaim queue 触发；两类事件不走 TCP。 |
| P0 | API/生命周期 | lane-local `BufferPool` 和 `BufferHandle` | V1 buffer 由库管理并绑定 lane；用户不能注册任意内存。 | `borrowBuffer()` 成功返回 lane-local buffer；completion 前复用或跨 lane 迁移 buffer 被检测为 API 误用或返回错误。 |
| P0 | Completion | Completion taxonomy | 将 completion 定义为本地事件流，至少区分 `RecvCompletion`、`ReclaimCompletion`、`ControlCompletion`，并为异步发送预留 `PostCompletion`。 | `pollCompletion()` 或等价 API 能区分 completion type、lane、slot/buffer、status 和 `user_data`；文档明确每类 completion 的来源和语义。 |
| P0 | API/发送 | 同步 `postSend()` | 第一迭代默认同步 publish：本端写 payload/header、commit slot、push ready notification 后返回。 | `postSend()` 返回 `Ok` 表示 publish 完成；返回 `QueueFull` / `NoBuffer` / `Error` 时不产生半发布消息；不要求用户 poll `PostCompletion` 才确认同步 publish。 |
| P0 | 正确性 | header integrity validation | header checksum 或等价完整性校验必选，保护 frame/slot 元数据。 | 注入 header corruption 时连接进入 protocol/data-plane error；错误 slot 不污染后续 slot 状态。 |
| P0 | Backpressure | 有界资源返回语义 | slot、buffer、ready queue、reclaim queue 都是有界资源，压力通过结构化结果返回。 | 覆盖 `NoBuffer`、`QueueFull`、`CompletionQueueFull`；这些状态不会被异常崩溃或无限阻塞替代。 |
| P0 | Progress | Core non-blocking poll | Core API 以 non-blocking poll 为基本 progress 机制。 | `pollCompletion(max_entries)` 或等价 poll API 在无事件时立即返回 empty；有事件时批量返回；错误时返回结构化 error。 |
| P1 | Progress/容量 | idle policy | V1 不能要求每个 idle connection 常驻 busy-poll CPU；progress 应按 lane 组织。 | benchmark 可选纯 `busy_poll`；可用性模式支持 `spin_then_sleep` 或等价 backoff；文档明确 interrupt/event 是后续硬件能力。 |
| P1 | API/异步 | 异步发送预留 | 第一迭代可以不实现 async submit queue，但 API/Completion 模型应允许后续 `submitSend()` + `PostCompletion`。 | 需求文档和 API 设计不把 `postSend()` 同步语义写死为唯一未来模式；`PostCompletion` 明确为本地 runtime 事件。 |
| P1 | Observability | 统计与诊断 | 输出足够解释性能和资源边界的 per-lane 指标。 | 至少记录 `lane_count`、slot 参数、completion 计数、NoBuffer/QueueFull 计数、close/error reason、payload checksum 开关。 |
| P1 | 测试 | 单元与集成测试 | 覆盖控制面 reject、slot commit、completion、buffer 生命周期和 backpressure。 | 本地测试能验证握手成功/失败、完整消息可读、reclaim 后可复用、completion 前复用失败、checksum 错误路径。 |
| P1 | Benchmark | 独立 benchmark | 提供不依赖 Thrift/Rocket 的 ping-pong、one-way 和 multi-lane scale-out benchmark。 | 报告 `lane_count=1,2,4,8,16` 的 latency、throughput、p99/p999、qps_per_total_core、CPU utilization、backpressure rate。 |
| P1 | Benchmark | high-pressure 探索点 | `32/64` lane 或高并发点必须报告，但不作为第一迭代硬门槛。 | 性能报告包含 `32,64` 探索点；结果单独标注 NUMA/SMT/core budget 风险，不混入 `1-16` gating 结论。 |
| P0 | Adapter | `netpoll + Kitex/Keitex` adapter | 通过 `netpoll` transport 抽象接入 ub.mem data plane，验证非 `fbthrift` 生态也能复用独立库。 | 至少跑通 unary echo 或等价最小 RPC；覆盖正确性、client/server lifecycle、socket baseline 对比和 adapter 侧错误日志。 |
| P0 | Adapter | `fbthrift + folly` adapter | 以 `folly` transport / `fbthrift` Rocket benchmark 路径为优先切入，验证 C++ RPC 栈 e2e。 | 至少跑通 Rocket unary request/response 或等价最小 benchmark；应用 payload 经 ub.mem data plane；保留 TCP/socket baseline 对比。 |
| P2 | Adapter | 生产级 adapter feature parity | V1 不覆盖完整 RPC transport 能力，只记录两个 proof adapter 与生产路径的差距。 | streaming、TLS、安全认证、service discovery、fallback、复杂 flow control 和完整异常恢复均列为后续；V1 artifact 必须记录缺口。 |
| P2 | 调度 | `LaneSelectionPolicy` 预留 | V1 默认 round-robin；后续可引入 queue-depth、payload-size、NUMA、priority、adapter shard key 等策略。 | `Channel` lane 选择不写死在不可替换实现里；复杂负载均衡不阻塞第一迭代。 |
| P2 | 内存模型 | arbitrary user memory registration | V1 不支持用户任意内存注册，只记录后续评估条件。 | 文档明确 user memory registration 非 V1 目标；后续触发条件包括 large payload、zero-copy serialization 或 adapter 证明需要。 |

## 5. 用户心智模型

V1 采用 hybrid API 心智：

- Core API 偏 verbs-like，显式表达 endpoint、connection、lane、buffer、completion 和 poll。
- `Channel` API 是薄 wrapper，降低 benchmark、V1 adapter 和后续 adapter 接入成本。

该设计不是 socket wrapper，也不是完整 URMA clone。内部语义贴近高性能用户态通信库，外部保留一个轻量 channel 接口，避免第一轮用户必须直接面对完整 verbs API。

## 6. 核心对象模型

### 6.1 Core API 对象

```text
Context
Endpoint
Connection
Lane
BufferPool
BufferHandle
SendRequest
Completion
Poller
```

对象职责如下：

| 对象 | V1 职责 |
|---|---|
| `Context` | 管理库级配置、日志、统计和资源初始化。 |
| `Endpoint` | 表示本端通信实体，负责 `listen` / `connect`、TCP bootstrap 和创建 `Connection`。 |
| `Connection` | 表示一对 peer 的通信关系，包含固定 `lane_count`、连接状态、错误信息和共享资源描述。 |
| `Lane` | 一条独立数据通路。每条 lane 有自己的 slot space、lane-local `BufferPool`、send/recv 元数据和 notification queue。 |
| `BufferPool` | 管理库分配的 ub.mem / shared-memory buffers。V1 不接受 arbitrary user memory registration。 |
| `BufferHandle` | 用户借出的 buffer，包含 data pointer、capacity、length、slot/buffer id、所属 lane 或 pool 信息。 |
| `SendRequest` | 描述一次发送，引用 `BufferHandle`，携带 `flags`、`user_data` 和可选 metadata。 |
| `Completion` | 本地完成事件抽象，按 type 区分 `RecvCompletion`、`ReclaimCompletion`、`ControlCompletion`，并为异步发送预留 `PostCompletion`。 |
| `Poller` | 轮询 completion、control event 和 error event。 |

### 6.2 Channel API

`Channel` 是薄 wrapper，不隐藏核心资源语义：

```text
Channel::borrowBuffer(size)
Channel::postSend(BufferHandle)
Channel::pollCompletion()
Channel::release(BufferHandle)
Channel::waitRecv(timeout)
Channel::waitCompletion(timeout)
```

`Channel` API 的约束：

- 不承诺 Core API 没有的语义，例如全局有序、无限 send queue、自动可靠恢复。
- 默认 lane 选择策略为 round-robin。
- Core API 允许调用方显式选择 lane。
- `wait-with-timeout` 只是 convenience helper，不是核心 progress 机制。

## 7. Buffer 模型

V1 使用库管理的 ub.mem / shared-memory buffer pool：

```text
borrowBuffer -> write payload -> postSend -> pollCompletion(ReclaimCompletion) -> reuse/release
```

要求：

- V1 默认使用 lane-local `BufferPool`。
- `BufferHandle` 在 borrowed / posted 生命周期内属于一个 lane。
- cross-lane buffer migration 不属于 V1。
- arbitrary user memory registration 不属于 V1。
- completion 前复用 `BufferHandle` 是 API 误用，测试必须覆盖该错误路径。

### 7.1 后续演进

lane-local `BufferPool` 是 V1 实现选择，不是长期调度模型限制。因为数据面暴露的是 bounded atomic message，而不是 byte stream，后续版本可以在 message 粒度引入 `LaneSelectionPolicy`，例如：

```text
round_robin
least_pending_slots
least_completion_lag
payload_size_aware
numa_aware
priority_aware
adapter_shard_key
hot_lane_avoidance
```

如果 benchmark、large payload 或 adapter use case 证明有必要，后续版本可以评估 arbitrary user memory registration。

## 8. 消息语义

V1 数据面定义为：

```text
bounded slot-based atomic message transport
```

核心语义：

- 每个 committed slot 表示一条完整消息。
- 未 commit 的 slot 对 receiver 不可见。
- API 层不承诺消息顺序。
- notification queue 的 FIFO 只表示 ready-slot event 顺序，不是应用消息顺序。
- 需要顺序的业务或 adapter 必须自带 sequence number，并在应用层或 adapter 层重排。
- queue、slot 和 buffer 都是有界资源。
- `NoBuffer` / `QueueFull` 是正常 backpressure，不是异常崩溃。

V1 不提供 TCP 式无界 byte stream，也不承诺内核 TCP 的拥塞控制、重传或自动恢复语义。

## 9. Slot / Commit 协议要求

发送路径：

```text
1. borrowBuffer(lane, size)
2. 用户写 payload
3. postSend(buffer, metadata)
4. sender 写入 frame metadata
5. sender publish / commit slot
6. sender 发送 ready-slot notification
7. pollCompletion() 返回 ReclaimCompletion 后 buffer 可复用或释放
```

接收路径：

```text
1. pollCompletion()
2. receiver 得到 committed complete message slot
3. 用户读取 payload
4. releaseRecv(buffer / slot)
5. slot 返回可用状态
```

frame metadata 至少包含：

```text
slot_id
lane_id
length
flags
message_type 或 user_metadata
commit / version marker
header checksum 或等价 header integrity validation
payload checksum flag
```

实现必须使用清晰的 publish/commit 边界，避免 receiver 观察到半写入 payload。推荐语义是 sender 在完成 payload 和 header 写入后，以 release 语义发布 commit marker；receiver 以 acquire 语义观察 commit 后读取完整消息。

## 10. Header 与 Payload 校验

V1 要求：

- header checksum 或等价 header integrity validation 必选。
- payload checksum 可选，可由配置或 per-message flag 控制。

处理要求：

| 情况 | V1 行为 |
|---|---|
| Header corruption | 库必须报告 protocol / data-plane error，并阻止错误 slot 继续污染连接状态。 |
| Payload checksum enabled 且校验失败 | 库报告 payload integrity error。 |
| Payload checksum disabled | payload 解析和业务级校验由应用或 adapter 负责。 |

该设计保护库自身依赖的 frame / slot 状态，同时避免强制每条业务 payload 支付校验成本。

## 11. TCP Bootstrap Handshake

V1 必须提供 TCP bootstrap control plane。TCP 用于跨机器建链、参数交换、错误通知和后续控制面扩展，不承载应用 payload。

### 11.1 角色

```text
Acceptor:
  listen(tcp_host, tcp_port)
  接收 initiator 的 bootstrap request
  校验协议版本、参数和资源能力
  返回 accept / reject

Initiator:
  connect(tcp_host, tcp_port)
  发送期望的数据面参数
  收到 accept 后 attach 数据面资源
```

### 11.2 握手字段

V1 握手至少交换：

```text
protocol_magic
protocol_version
role
peer_id
connection_id
lane_count
slot_size
slots_per_lane
buffer_pool_size
max_message_size
data_plane_type: ub_mem / shm
shared_region_descriptor
notification_queue_descriptor
feature_bits
ordering_mode: unordered_atomic_message
failure_model: memory_domain_assumed_reliable
```

### 11.3 Reject 条件

V1 必须 reject：

```text
protocol_version 不兼容
lane_count 不匹配或超过本端能力
slot_size / max_message_size 不支持
shared region 无法 attach
notification queue 无法 attach
feature_bits 要求本端不支持的必选特性
peer_id / connection_id 冲突
```

### 11.4 状态迁移

成功路径：

```text
TcpConnected
HandshakeSent
HandshakeAccepted
DataPlaneAttached
Ready
```

失败路径：

```text
Rejected
AttachFailed
ProtocolError
Closed
```

## 12. TCP Control Connection 生命周期

V1 握手完成后保持 TCP control connection 打开。

V1 必须支持的最小控制事件：

```text
graceful close
protocol error notification
data-plane attach failure notification
peer-side reject reason
optional heartbeat / liveness probe
```

后续可扩展控制事件：

```text
fallback / downgrade negotiation
lane health notification
flow-control tuning
feature renegotiation
diagnostic snapshot request
adapter-specific control messages
```

边界要求：

- TCP control connection 不是 V1 的 data fallback path。
- 应用 payload 不得静默 fallback 到 TCP。
- 如果未来版本支持 TCP payload fallback，必须通过 feature negotiation 显式启用，并在 benchmark 报告中单独标记。

## 13. Lane 模型

V1 支持 fixed multi-lane：

- `lane_count` 在 TCP bootstrap 中协商。
- 每条 lane 拥有独立 slot space、lane-local `BufferPool` 和 notification queue。
- V1 不支持运行时增减 lane。
- V1 不支持运行时迁移 outstanding messages。

V1 gating 的主要范围为：

```text
lane_count = 1,2,4,8,16
```

`lane_count = 32,64` 必须出现在性能报告中，但不作为 V1 是否通过的硬门槛。

## 14. Progress 模型

Core API 提供 non-blocking poll：

```text
pollCompletion(max_entries)
```

该 API 必须立即返回：

```text
有事件 -> 返回 completions / events
无事件 -> 返回 empty
发生错误 -> 返回结构化 error
```

`Channel` wrapper 可以提供：

```text
waitRecv(timeout)
waitCompletion(timeout)
drainCompletions()
```

要求：

- wait-with-timeout 是 convenience，不是核心 progress 机制。
- 默认实现可以 spin 一段时间后 sleep / backoff。
- benchmark 必须能选择纯 busy-poll 模式，避免调度策略污染性能结果。

## 15. 错误模型

V1 将错误分为以下类别。

### 15.1 Backpressure

```text
NoBuffer
QueueFull
CompletionQueueFull
ReceiverNotReleasing
```

Backpressure 是正常返回值，调用方可以重试、等待 completion 或降低发送压力。

### 15.2 Protocol / Data-plane Error

```text
InvalidHeader
HeaderChecksumError
PayloadChecksumError
InvalidSlotState
UnsupportedFeature
MalformedFrame
```

Protocol / data-plane error 使 `Connection` 进入 error state，并通过 TCP control connection 尽量通知 peer。

### 15.3 Control-plane Error

```text
HandshakeRejected
VersionMismatch
TcpControlClosed
HeartbeatTimeout
```

control connection close 或 heartbeat timeout 使 `Connection` 进入 closed / error state。

### 15.4 Resource / Attach Error

```text
SharedRegionAttachFailed
NotificationQueueAttachFailed
InsufficientLanes
SlotSizeUnsupported
```

这类错误发生在 handshake 或 attach 阶段，必须带 reject / error reason。

### 15.5 Peer / Application Lifecycle

```text
GracefulClose
PeerClosed
LocalClose
```

V1 支持 graceful close 语义，但不承诺从 peer crash、machine crash 或 ub 硬件不可用中恢复。

## 16. Failure Model

V1 的可靠性假设是：

```text
memory-domain assumed reliable under healthy hardware
```

含义：

- 在硬件健康、shared mapping 有效、协议状态未损坏的条件下，committed message 被视为可由 peer 轮询到。
- V1 不做 retransmission。
- V1 不做 ACK repair。
- V1 不做 path failover。
- V1 不保证检测所有 peer crash、silent memory corruption 或硬件级 ub failure。
- 硬件不可用、机器崩溃、ub fabric 失效属于 V1 软件恢复范围之外。

业务 payload 的语义正确性仍由业务或 adapter 负责。例如 payload 解析失败、业务 checksum 失败或应用层 sequence gap，应由上层处理。

## 17. 结构化 Result

V1 API 不应使用异常表达正常 backpressure。推荐所有 `borrow`、`post`、`poll`、`release` API 返回结构化结果：

```text
Ok(value)
Empty
Backpressure(code)
Error(code, reason)
```

要求：

- `NoBuffer` / `QueueFull` 可被调用方显式区分。
- `Connection` 的 error state 可查询。
- control-plane reject / error reason 可被记录到日志和 benchmark artifact。

### 20.3 Benchmark 矩阵

V1 gating 点：

```text
lane_count = 1,2,4,8,16
```

报告必须包含探索点：

```text
lane_count = 32,64
```

解释边界：

- `1,2,4,8,16` 是 V1 主要验收范围。
- `32,64` 用于观察高并发压力、NUMA/SMT、core budget 和资源模型变化，不作为 V1 通过/失败硬门槛。

建议固定参数：

```text
slot_size = 小消息默认值 + 一个中等 payload 档位
slots_per_lane = 固定值
payload_checksum = off / on
progress_mode = busy_poll
```

对比基线：

```text
TCP socket baseline
现有 shm_poller / shm_directpoll benchmark 经验
```

独立 benchmark 承担 `1,2,4,8,16` 的完整性能 gating。e2e adapter 承担真实 RPC 链路说服力 gating，不要求覆盖同等完整矩阵。

### 20.4 E2E adapter 验收

V1 adapter 验收目标不是完整替代生产 RPC transport，而是证明真实 RPC 栈可以通过 ub.mem data plane 完成端到端请求。每个 adapter 至少覆盖：

```text
client/server 真实进程启动
TCP bootstrap/control connection 建链
ub.mem data plane 承载 request payload
ub.mem data plane 承载 response payload
unary request/response 正确性
socket baseline 对比
adapter error / close 路径可观测
```

第一迭代选定两条 adapter 路径：

```text
netpoll + Kitex/Keitex
fbthrift + folly
```

每条 adapter 至少跑通 `lane_count=1` 和一个 multi-lane 点，建议选择 `lane_count=4` 或 `lane_count=8`。完整 `1,2,4,8,16` 性能矩阵仍由独立 benchmark 承担；adapter 报告重点是 e2e 可用性、集成代价、RPC 栈改造点和相对 socket baseline 的方向性收益。

## 23. 决策摘要

| 决策 | V1 结论 |
|---|---|
| 用户心智 | hybrid：Core verbs-like，Channel thin wrapper。 |
| 第一消费者 | 独立库和 benchmark 是主交付；`netpoll + Kitex/Keitex`、`fbthrift + folly` 是最小 e2e proof。 |
| adapter 范围 | V1 实现两个最小 unary request/response proof，不追求生产级 RPC transport feature parity。 |
| 控制面 | TCP bootstrap，握手后保持 control connection。 |
| 数据面 | ub.mem / shared-memory user-space path。 |
| lane | fixed multi-lane，V1 不做 dynamic lane。 |
| buffer | lane-local library-managed `BufferPool`。 |
| 用户内存注册 | V1 不支持，后续评估。 |
| 消息语义 | bounded slot-based atomic message，API 层 unordered。 |
| ordering | 不承诺 FIFO 或全局有序，需要顺序的上层自带 sequence number。 |
| checksum | header 必选，payload 可选。 |
| progress | Core non-blocking poll，Channel 可提供 wait-with-timeout。 |
| reliability | healthy hardware 下 memory-domain assumed reliable，不做 retransmission。 |
| benchmark gating | `lane_count=1,2,4,8,16`。 |
| high-pressure 报告 | `32,64` 必须报告，不作为硬门槛。 |
